Terminar la pantalla de designación por carrera (con pestañas y barra de progreso) y

## Qué se hizo
- Se cerró la pantalla de designación dentro de una carrera específica: se eligen
  gestión y periodo, y se ve la lista de materias organizadas en pestañas ("por asignar",
  "asignadas", "resumen de designaciones"), con un anillo de progreso que muestra cuántas
  materias ya tienen docente asignado.
- Se escribió `design.md`: un documento con las reglas visuales obligatorias del sistema
  (colores, tipografía, cómo se ven los botones/tablas/badges, qué está prohibido —
  por ejemplo, ningún degradado de color). A partir de ahora toda pantalla nueva debe
  seguir esas reglas.
- Se preparó el terreno para poder anotar la carga horaria de cada materia: se agregó el
  campo "horas" a materias (todavía sin usarse en ningún cálculo, eso es la Fase 3).
- Se cambió el comportamiento de borrado de los catálogos: antes, borrar una carrera
  borraba en cadena todas sus materias, grupos y designaciones sin avisar. Ahora el
  sistema bloquea el borrado si hay datos dependientes, y muestra un mensaje explicando
  por qué no se pudo borrar.
- Se construyeron las primeras 6 pantallas de catálogos, cada una con su listado, su
  formulario de creación/edición y su borrado seguro:
  - **Carreras** (sigla, nombre).
  - **Materias** (sigla, nombre, carrera, horas) — con filtro por carrera.
  - **Grupos** (materia, código, estado habilitado/deshabilitado) — con filtro por
    materia y estado.
  - **Docentes** (nombre, CI, carrera de origen) — con buscador por nombre o CI.
  - **Gestiones** (años académicos).
  - **Periodos** (1, 2, Verano).
- Se agregó una sección "Catálogos" en el menú lateral, con acceso directo a las 6
  pantallas nuevas.
- Se rellenó retroactivamente la bitácora del 2026-07-10, que tenía una sesión de trabajo
  (login, asignación masiva, copiar designaciones, pantalla por carrera) sin documentar.

## Decisiones tomadas
- Antes de construir las 6 pantallas de catálogos, se extrajeron a componentes
  reutilizables varias piezas visuales que estaban duplicadas (íconos, selector
  desplegable, paginación, estado vacío) — evita que cada pantalla nueva reinvente lo
  mismo y mantiene el look consistente en todo el sistema.
- De paso se corrigió un detalle que se había colado en la tabla de "designaciones por
  carrera": un ícono tenía una clase de degradado de color aplicada, algo explícitamente
  prohibido por `design.md`. Se quitó.
- El borrado de catálogos se implementó verificando primero si hay datos dependientes
  (en vez de dejar que la base de datos tire un error técnico), para mostrar un mensaje
  entendible como "no se puede eliminar: tiene materias asociadas".
- El código de grupo (ej. "A", "B") solo tiene que ser único dentro de la misma materia,
  no en todo el sistema — dos materias distintas pueden tener ambas un "Grupo A".
- Se avanzó fase por fase con commits separados por cada pieza (migraciones, componentes
  compartidos, y un commit por cada catálogo) en vez de un commit único al final, para
  que el historial quede ordenado y cada paso se pueda revisar o revertir por separado.
