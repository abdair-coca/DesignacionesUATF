# Bitácora — 2026-07-11

## Objetivo del día
Terminar la pantalla de designación por carrera (con pestañas y barra de progreso) y
arrancar la Fase 2 del roadmap: catálogos con pantallas reales (carreras, materias,
grupos, docentes, gestiones, periodos), que hoy solo existen por datos de prueba.

## Qué se hizo
- Se cerró la pantalla de designación dentro de una carrera específica: se eligen
  gestión y periodo, y se ve la lista de materias organizadas en pestañas ("por asignar",
  "asignadas", "resumen de designaciones"), con un anillo de progreso que muestra cuántas
  materias ya tienen docente asignado.
- Se escribió `design.md`: un documento con las reglas visuales obligatorias del sistema
  (colores, tipografía, cómo se ven los botones/tablas/badges, qué está prohibido —
  por ejemplo, ningún degradado de color). A partir de ahora toda pantalla nueva debe
  seguir esas reglas.
- Se preparó el terreno para poder anotar la carga horaria de cada materia: se agregó el
  campo "horas" a materias (todavía sin usarse en ningún cálculo, eso es la Fase 3).
- Se cambió el comportamiento de borrado de los catálogos: antes, borrar una carrera
  borraba en cadena todas sus materias, grupos y designaciones sin avisar. Ahora el
  sistema bloquea el borrado si hay datos dependientes, y muestra un mensaje explicando
  por qué no se pudo borrar.
- Se construyeron las primeras 6 pantallas de catálogos, cada una con su listado, su
  formulario de creación/edición y su borrado seguro:
  - **Carreras** (sigla, nombre).
  - **Materias** (sigla, nombre, carrera, horas) — con filtro por carrera.
  - **Grupos** (materia, código, estado habilitado/deshabilitado) — con filtro por
    materia y estado.
  - **Docentes** (nombre, CI, carrera de origen) — con buscador por nombre o CI.
  - **Gestiones** (años académicos).
  - **Periodos** (1, 2, Verano).
- Se agregó una sección "Catálogos" en el menú lateral, con acceso directo a las 6
  pantallas nuevas.
- Se rellenó retroactivamente la bitácora del 2026-07-10, que tenía una sesión de trabajo
  (login, asignación masiva, copiar designaciones, pantalla por carrera) sin documentar.

## Decisiones tomadas
- Antes de construir las 6 pantallas de catálogos, se extrajeron a componentes
  reutilizables varias piezas visuales que estaban duplicadas (íconos, selector
  desplegable, paginación, estado vacío) — evita que cada pantalla nueva reinvente lo
  mismo y mantiene el look consistente en todo el sistema.
- De paso se corrigió un detalle que se había colado en la tabla de "designaciones por
  carrera": un ícono tenía una clase de degradado de color aplicada, algo explícitamente
  prohibido por `design.md`. Se quitó.
- El borrado de catálogos se implementó verificando primero si hay datos dependientes
  (en vez de dejar que la base de datos tire un error técnico), para mostrar un mensaje
  entendible como "no se puede eliminar: tiene materias asociadas".
- El código de grupo (ej. "A", "B") solo tiene que ser único dentro de la misma materia,
  no en todo el sistema — dos materias distintas pueden tener ambas un "Grupo A".
- Se avanzó fase por fase con commits separados por cada pieza (migraciones, componentes
  compartidos, y un commit por cada catálogo) en vez de un commit único al final, para
  que el historial quede ordenado y cada paso se pueda revisar o revertir por separado.

## Qué se hizo (sesión 2, Fase 3 — carga horaria y choques)
- Se creó un servicio (`CargaAcademicaService`) que calcula, para un docente en una
  gestión y periodo dados, cuántas horas ya tiene asignadas sumando las materias de sus
  designaciones activas. El mismo servicio detecta si un grupo ya tiene otro docente
  asignado en esa misma gestión y periodo (choque de horario).
- Se conectó ese cálculo al formulario de crear/editar designación: a medida que se va
  eligiendo docente, materia, grupo, gestión y periodo, aparece automáticamente (sin
  recargar la página) un aviso si el docente quedaría con más de 6 horas, o si el grupo
  elegido ya tiene otra designación activa. Es solo una advertencia — no impide guardar,
  porque el supervisor no definió todavía si esto debe bloquear o no.
- Se reemplazó la verificación de "grupo ya asignado" que estaba repetida (con el mismo
  código) tanto en la asignación masiva por carrera como en copiar designaciones, para
  que las tres pantallas (individual, masiva, copiar) usen la misma lógica.
- De paso, las pantallas de crear/editar designación (las últimas dos que quedaban con el
  estilo visual viejo) se actualizaron para cumplir `design.md` (selector desplegable con
  flecha propia en vez del feo nativo del navegador, botones planos).

## Decisiones tomadas (Fase 3)
- La advertencia de 6 horas se calcula sobre el total "proyectado": horas que el docente
  ya tiene asignadas en esa gestión/periodo, más las horas de la materia que se está por
  asignar ahora mismo. Así el aviso aparece antes de guardar, no después.
- Al editar una designación existente, el cálculo se hace excluyendo esa misma
  designación (si no, el sistema avisaría de un choque contra sí misma todo el tiempo).
  Se verificó manualmente que esto funciona antes de dar el paso por cerrado.

## Qué se hizo (sesión 3, Fase 4 — filtros en el listado)
- El listado plano de designaciones (al que se llega, por ejemplo, desde "Ver
  designaciones" dentro de una carrera) ahora se puede filtrar por carrera, gestión,
  periodo y estado al mismo tiempo, todo sin recargar la página completa.
- Se creó una barra de filtros reutilizable (`FilterBar`) pensada para no tener que
  reconstruirla cuando llegue el dashboard de Fase 5.

## Decisiones tomadas (Fase 4)
- El filtro por carrera en este listado reemplaza la lógica anterior, que solo mostraba
  la carrera si se llegaba desde un link específico (quedaba fija, no se podía cambiar).
  Ahora es un filtro más, como los otros tres — se puede limpiar o cambiar libremente.

## Qué se hizo (sesión 4, Fase 5 — dashboard)
- Se creó la pantalla de Dashboard, y ahora es la primera pantalla que se ve al iniciar
  sesión (antes era "Designación de Docentes"). Muestra, para la gestión y periodo
  elegidos (por defecto la gestión más reciente y el primer periodo):
  - Cuántos grupos habilitados todavía no tienen ningún docente asignado, con un botón
    directo para designar cada uno.
  - Cuántas designaciones hay en cada estado (propuesta, aprobada, rechazada).
  - Qué docentes están por debajo de las 6 horas recomendadas, y cuántas horas tienen
    hasta ahora.
- Se agregó "Dashboard" como primer ítem del menú lateral.

## Decisiones tomadas (Fase 5)
- Se extrajeron dos piezas visuales más a componentes compartidos antes de construir el
  dashboard: la tarjeta de número grande (KPI) que ya se usaba en la pantalla de
  designaciones por carrera, y se reusó la barra de filtros creada en Fase 4 — el
  dashboard no inventó nada visual nuevo, todo ensamblado con piezas ya existentes.
- El límite de 6 horas (que hasta ahora vivía solo dentro del formulario de designación)
  se movió al servicio de carga académica, para que tanto el formulario como el
  dashboard lean el mismo número en vez de tener cada uno el suyo.
