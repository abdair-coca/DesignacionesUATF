# Bitácora — 2026-07-10

## Objetivo del día
Cambiar el stack del proyecto por decisión de arriba: usar React (dentro del mismo Laravel,
sin separar en dos proyectos) para priorizar la experiencia de usuario, y pasar de SQLite a
PostgreSQL para que el ambiente de desarrollo se parezca al de producción.

## Qué se hizo
- Se instaló PostgreSQL como base de datos de desarrollo, reemplazando SQLite. Se creó una
  base de datos propia para este proyecto (`designaciones_uatf`) en el servidor Postgres
  local, separada de otras bases de datos que ya existían en ese mismo servidor.
- Se migró todo el esquema (las 9 tablas del sistema) a PostgreSQL sin cambios de diseño —
  el modelo de datos es el mismo, solo cambia el motor.
- Se agregó Inertia.js: esto permite que Laravel siga manejando las rutas y la lógica del
  servidor como siempre, pero que la interfaz esté hecha en React en vez de plantillas Blade.
  No es una aplicación separada ni una API — sigue siendo un solo proyecto, un solo despliegue.
- Se reconstruyó la sección "Designación de Docentes" completa en React: listado con
  paginación, formulario de creación/edición y vista de historial, con el mismo
  comportamiento que tenía antes (incluyendo el registro automático de cambios en el
  historial), pero con una interfaz más moderna e interactiva.
- Se probó el flujo completo contra el servidor real (no solo pruebas internas): se levantó
  el servidor, se armó una petición real de creación de designación tal como la haría el
  navegador (con las cookies y el token de seguridad que usa Inertia), y se confirmó que la
  designación quedó guardada correctamente en la base de datos.
- Se actualizó `CLAUDE.md` y `README.md` para reflejar el stack nuevo.

## Decisiones tomadas
- Para conectar React con Laravel se eligió **Inertia.js** (y no una API REST separada, ni
  "islas" de React sobre Blade). Esto se confirmó directamente antes de empezar: mantiene
  todo en un solo repositorio y despliegue, sin duplicar lógica de autenticación ni armar
  una capa de API que después haya que mantener aparte.
- Los estilos se armaron con Tailwind CSS, que es lo que ya trae Laravel por defecto en este
  scaffold. No se agregó Bootstrap para no meter dos sistemas de estilos distintos mientras
  no llega el template Color Admin — cuando llegue, su HTML/CSS se va a portar a componentes
  React, no a Blade.
- Se creó una base de datos Postgres nueva y separada (`designaciones_uatf`) en vez de
  reusar `colab_db`, porque esa base ya tenía tablas de otro proyecto (categorías, notas,
  usuarios) y correr las migraciones ahí arriesgaba pisar datos que no son nuestros. Se
  confirmó esta decisión antes de crear nada.
- La extensión de PHP para PostgreSQL ya estaba habilitada en este equipo (no hizo falta
  tocar `php.ini` esta vez, a diferencia de SQLite el primer día).

## Qué se hizo (sesión 2, más tarde el mismo día)
- Se agregó login básico: pantalla de inicio de sesión, y desde ese momento todo el sistema
  quedó protegido — hace falta iniciar sesión para ver cualquier pantalla. Esto también
  permite registrar quién crea o modifica cada designación (antes ese dato quedaba vacío).
- Se agregaron dos formas nuevas de cargar designaciones, pensadas para no tener que cargar
  materia por materia en universidades con muchas carreras:
  - **Asignación masiva por carrera**: se elige carrera + gestión + periodo y se ve una lista
    de todos los grupos sin designación, con un selector de docente al lado de cada uno —
    se guardan todas juntas de una vez.
  - **Copiar designaciones de otra gestión**: útil porque las designaciones no cambian mucho
    de un año a otro. Se elige carrera + gestión/periodo de origen y destino, se ve una
    vista previa editable, y se salta automáticamente cualquier grupo que ya tenga
    designación en el destino (no pisa nada existente).
- Se rediseñó la pantalla principal de designaciones: ahora se organiza por carrera, con un
  resumen general (cuántas carreras tienen designaciones activas, pendientes o ninguna) y
  filtros por gestión, periodo y estado.

## Decisiones tomadas (sesión 2)
- Las dos funciones nuevas (asignación masiva y copiar) no estaban en el pedido original del
  día, pero se agregaron porque cargar designación por designación no es práctico para toda
  la universidad. Se construyeron sobre el mismo patrón de controllers ya usado, sin tocar
  el modelo de datos.
- El sidebar se mantuvo con un solo ítem visible ("Designación de Docentes"); las dos
  funciones nuevas se ofrecen como botones dentro de esa sección, no como ítems nuevos en el
  menú lateral — pedido explícito del dueño de no hacer crecer el menú todavía.
