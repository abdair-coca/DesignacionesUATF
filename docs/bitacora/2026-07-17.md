# Bitácora — 2026-07-17

## Objetivo del día
Implementar flujo de aprobación de designaciones por carrera (rol admin, revisión en lote, previsualización al pegar) y hacer configurable el límite de horas.

## Qué se hizo
- **Migraciones**: Se agregó columna `is_admin` a `users` y se creó tabla `revisiones` para trackear ciclos de revisión por carrera+gestion+periodo.
- **Modelos**: Se creó `Revision.php` con relaciones a carrera, usuario, gestión y periodo. Se actualizó `User.php` con `is_admin`.
- **Límite configurable**: Se creó `config/designaciones.php` que lee `DESIGNACIONES_LIMITE_HORAS` de `.env`. `CargaAcademicaService::getLimite()` reemplaza la constante hardcodeada en todos los controladores y servicios.
- **RevisionController**: 5 métodos — `solicitar` (usuario envía carrera a revisión), `pendientes` (admin ve lista), `revisar` (admin ve tabla de designaciones), `procesar` (admin aprueba/rechaza en lote), `completar` (admin cierra la revisión). Los métodos de admin verifican `is_admin`.
- **Rutas**: 5 rutas nuevas para el flujo de revisión + 1 para previsualizar pegado.
- **Página Pendientes.jsx**: Lista para admin de carreras esperando revisión, con datos de solicitante, fecha, gestión/periodo.
- **Página Revisar.jsx**: Tabla con todas las designaciones de la carrera, botones individuales y en lote para aprobar/rechazar, y botón "Completar revisión" que registra quién y cuándo.
- **Carrera.jsx**: Sidebar ahora muestra estado de revisión (Sin enviar / En revisión / Revisado) con badge de color. Botón "Enviar a revisión" para usuarios no-admin. Se crea revision y se evitan duplicados pendientes.
- **PasteModal previsualización**: Nuevo endpoint `POST /designaciones/previsualizar-pegado` que valida filas sin crearlas. El modal muestra columna "Estado" con iconos verde (Listo) o ámbar (Grupo ocupado / Excede horas). Checkboxes deshabilitados para filas saltadas. Resumen "N se pegarán, X se omitirán".
- **Seeder**: Usuario admin agregado (`admin@uatf.edu.bo`).
- **Tests**: 7 tests nuevos para el flujo de revisión (enviar, duplicados, aprobar lote, rechazar lote, completar, permisos). 65/65 tests pasan.

## Decisiones tomadas
- **Tabla `revisiones` separada** en vez de campo `en_revision` en designaciones: Permite trackear el ciclo completo (solicitante, fecha, revisor, fecha revisión) sin contaminar la tabla principal. La unique key compuesta `(carrera_id, Id_gestion, Id_periodo, estado)` evita dos revisiones pendientes simultáneas para el mismo contexto.
- **Admin como flag booleano** en lugar de tabla roles: Simple para el caso actual, un solo admin general. En el futuro se puede migrar a sistema de roles.
- **Previsualización como endpoint separado** de `pegar()`: Sigue el principio de responsabilidad única. El modal primero valida, muestra resultados, luego el usuario decide pegar.
- **Límite de horas por .env**: Permite cambiar sin migraciones. El valor default 6 mantiene compatibilidad con el comportamiento anterior.

## Bloqueos / dudas para el supervisor
- El flujo de revisión está implementado para un admin general. Cuando se definan los roles por carrera (jefe de carrera), habrá que filtrar qué carreras puede revisar cada admin.
- El PasteModal usa un endpoint separado de previsualización. Si hay muchas filas (ej. 50+), la experiencia podría ser lenta por las 2 llamadas (preview + pegar). Se puede optimizar uniendo ambos en una sola llamada si es necesario.
